与智能体协作，为智能体编写高效工具

Writing effective tools for agents — with agents | Anthropic | 2025-09-11

智能体的效能取决于我们提供给它们的工具。我们将分享如何编写高质量的工具与评测，以及如何通过使用 Claude 让它为自身优化工具以提升性能。

Model Context Protocol (MCP) 可以赋予 LLM 智能体多达数百种工具来解决真实世界的任务。但我们怎样才能让这些工具发挥最大的效果呢？

本文介绍我们在各种智能体 AI 系统1中提升性能的最有效技巧。

我们首先将介绍你可以如何：

  • 构建并测试工具原型
  • 与智能体一起创建并运行全面的工具评测
  • 与 Claude Code 等智能体协作，自动提升工具的性能

最后，我们总结了在此过程中识别出的高质量工具编写关键原则：

  • 选择哪些工具应该实现（以及哪些不该实现）
  • 通过命名空间为工具划定清晰的功能边界
  • 让工具向智能体返回有意义的上下文
  • 优化工具响应的 token 效率
  • 对工具描述和规格进行提示工程

什么是工具？

在计算领域，确定性系统在给定相同输入时每次都产生相同输出，而非确定性系统（如智能体）即使在相同的起始条件下也可能生成不同的响应。

在传统的软件编写中，我们是在确定性系统之间建立契约。例如，像 getWeather(“NYC”) 这样的函数调用每次都会以完全相同的方式获取纽约市的天气。

工具是一类新的软件，它体现的是确定性系统与非确定性智能体之间的契约。当用户询问“我今天需要带伞吗？”，智能体可能会调用天气工具，也可能凭借常识作答，或先提出一个关于位置的澄清问题。有时，智能体还可能产生幻觉，甚至不理解如何使用某个工具。

这意味着在为智能体编写软件时需要彻底重新思考：我们不再像为其他开发者或系统编写函数和 API 那样编写工具和 MCP 服务器，而是要专门为智能体设计它们。

我们的目标是通过工具让智能体能够在更广泛的任务上发挥作用，采用多种成功策略。幸运的是，根据我们的经验，对智能体最“符合人体工学”的工具往往同样直观易于人类理解。

如何编写工具

本节将介绍如何与智能体协作来编写并改进你提供给它们的工具。先快速搭建工具原型并在本地测试，随后运行全面的评测来衡量后续改动。与智能体协同工作，你可以反复评估和改进工具，直到智能体在真实任务中表现出色。

构建原型

如果不亲自上手，很难预判智能体觉得哪些工具顺手、哪些不顺手。先快速搭建工具原型。如果你使用 Claude Code（甚至一次性）来编写工具，最好向 Claude 提供工具所依赖的软件库、API 或 SDK（包括 MCP SDK）的文档。适合 LLM 的文档通常可以在官方文档站点的扁平 llms.txt 文件中找到（例如我们的 API 文档）。

将工具封装在 本地 MCP 服务器 或 桌面扩展（DXT）中，可以让你在 Claude Code 或 Claude 桌面应用里连接并测试这些工具。

要在 Claude Code 中连接本地 MCP 服务器，请运行 claude mcp add <name> <command> [args...]。

要在 Claude 桌面应用中连接本地 MCP 服务器或 DXT，请分别前往 Settings > Developer 或 Settings > Extensions。

工具也可以直接传入 Anthropic API 调用以便进行程序化测试。

亲自测试工具，找出任何粗糙之处。收集用户反馈，培养对你希望工具支持的用例和提示的直觉。

运行评测

接下来，你需要通过运行评测来衡量 Claude 使用你的工具的效果。首先生成大量基于真实世界使用场景的评测任务。我们建议与智能体协作，帮助分析结果并确定如何改进工具。在我们的 工具评测手册 中可以看到整个流程。

生成评测任务

借助早期原型，Claude Code 可以迅速探索你的工具并创建数十个提示与响应对。提示应源于真实世界的使用场景，并基于真实的数据源和服务（例如内部知识库和微服务）。我们建议避免过于简单或表面的“沙盒”环境，因为它们无法对你的工具进行充分复杂度的压力测试。高质量的评测任务可能需要多个工具调用，甚至数十次。

以下是一些高质量任务示例：

  • 安排下周与 Jane 的会议讨论我们最新的 Acme Corp 项目，附上上次项目规划会议的记录，并预定一个会议室。
  • 客户 ID 9182 报告称一次购买尝试被多收了三次费用。查找所有相关日志条目，并确定是否有其他客户受到同样问题影响。
  • 客户 Sarah Chen 刚提交了取消请求。拟定一个挽留方案，需要确定：(1) 她离开的原因，(2) 哪种挽留方案最具吸引力，(3) 在提出报价前我们需要留意的任何风险因素。

以下是一些较弱的任务示例：

  • 安排下周与 jane@acme.corp 的会议。
  • 在支付日志中搜索 purchase_complete 和 customer_id=9182。
  • 找到客户 ID 45892 的取消请求。

每个评测提示都应配有可验证的响应或结果。验证器可以简单到对比基准答案与采样响应的字符串是否完全一致，也可以像请 Claude 进行判定那样复杂。避免过于严格的验证器，因为它们可能会因格式、标点或同义表达等表面差异而拒绝正确答案。

对于每个提示-响应对，你还可以选择指定希望智能体调用的工具，以衡量智能体在评测中是否真正理解每个工具的用途。然而，鉴于正确完成任务通常存在多种途径，务必避免过度规定或过拟合策略。

执行评测

我们建议通过直接调用 LLM API 的方式以程序化形式运行评测。使用简单的智能循环（while 循环交替包裹 LLM API 调用和工具调用）：每个评测任务对应一个循环。每个评测智能体应获得一个任务提示以及你的工具。

在评测智能体的系统提示中，我们建议要求它们不仅输出结构化的响应块（用于验证），还要输出推理和反馈块。要求它们在工具调用和响应块之前输出这些内容，可以通过触发链式思维（CoT）行为来提升 LLM 的有效智能。

如果你在使用 Claude 运行评测，可以开启 interleaved thinking，即可“开箱即用”类似能力。这有助于你探究智能体为何（不）调用某些工具，并指出工具描述和规格中需要改进的具体部分。

除了总体准确率之外，我们建议收集其他指标，例如单个工具调用和任务的总运行时间、工具调用总次数、token 总消耗量以及工具错误。跟踪工具调用可以揭示智能体常用的工作流，并为工具整合提供机会。

分析结果
智能体是你发现问题和提供反馈的得力伙伴，无论是互相矛盾的工具描述、低效的工具实现，还是令人困惑的工具架构。需要注意的是，智能体在反馈和响应中遗漏的内容往往比它们提到的更重要。LLM 并不总是会言出必行。

观察智能体在哪些地方被难住或感到困惑。阅读评测智能体的推理和反馈（或 CoT）以找出棱角分明之处。审查原始对话记录（包括工具调用和工具响应），捕捉任何未在智能体 CoT 中明确描述的行为。读懂弦外之音；记住你的评测智能体未必知道正确答案和策略。

分析工具调用指标。如果出现大量冗余的工具调用，可能意味着需要重新调整分页或 token 限制参数；如果大量出现参数无效的工具错误，则可能需要提供更清晰的工具描述或更好的示例。当我们发布 Claude 的网页搜索工具时，我们发现 Claude 会无端在工具的 query 参数后附加 2025，从而偏离搜索结果并降低表现（我们通过改进工具描述把 Claude 引导回正轨）。

与智能体协作

你甚至可以让智能体为你分析结果并改进工具。只需将评测智能体的对话记录拼接起来，粘贴到 Claude Code 中即可。Claude 擅长分析对话记录并一次性重构大量工具，例如在出现新的改动时，确保工具实现与描述保持一致。

事实上，本文中的大部分建议都来自我们反复使用 Claude Code 优化内部工具实现的过程。我们的评测建立在内部工作区之上，反映了真实工作流程的复杂性，包括真实的项目、文档和消息。

我们依靠留出的测试集确保不会在“训练”评测上过拟合。这些测试集表明，即便是“专家”级工具实现（无论是由研究人员手写，还是由 Claude 自动生成），我们仍能进一步提升性能。

下一节，我们将分享在这一过程中学到的一些经验。

编写高效工具的原则

本节我们将这些经验提炼成编写高效工具的若干指导原则。

为智能体选择合适的工具

更多的工具不一定意味着更好的结果。我们常见的一个错误是，仅仅把现有软件功能或 API 端点包裹成工具——无论这些工具是否适合智能体。这是因为智能体与传统软件具有不同的“可供性”，也就是说，它们理解可供采取的行动的方式不同。

LLM 智能体的“上下文”是有限的（即一次能处理的信息量受限），而计算机内存却廉价而充裕。想象在通讯录中搜索联系人这一任务。传统软件可以高效地逐个存储并处理联系人，在前进之前检查每一项。

然而，如果 LLM 智能体使用的工具返回了所有联系人，然后不得不逐 token 阅读每一个条目，它就会把有限的上下文空间浪费在无关信息上（想象你在通讯录里逐页从上到下阅读，进行蛮力搜索）。更好且更自然的做法（对智能体和人类都一样）是先跳转到相关页面（例如按字母顺序查找）。

我们建议构建少量经过深思熟虑的工具，针对特定的高价值工作流，并与评测任务保持一致，然后逐步扩展。在通讯录的例子中，你可以考虑实现 search_contacts 或 message_contact 工具，而不是 list_contacts。

工具可以整合功能，在后台处理可能多个离散操作（或 API 调用）。例如，工具可以为响应补充相关元数据，或通过一次工具调用处理常见的链式多步任务。

以下是一些示例：

  • 与其分别实现 list_users、list_events 和 create_event 工具，不如实现一个 schedule_event 工具，用于查找空闲时间并安排事件。
  • 与其实现 read_logs 工具，不如实现 search_logs 工具，只返回相关的日志行及其上下文。
  • 与其实现 get_customer_by_id、list_transactions 和 list_notes 工具，不如实现 get_customer_context 工具，一次性汇总客户近期且相关的信息。

确保你构建的每个工具都有明确而独特的目标。工具应帮助智能体像人类在拥有同样资源时那样，将任务拆解并完成，同时减少中间输出本会消耗的上下文。

过多或功能重叠的工具也会分散智能体追求高效策略的注意力。对将要构建（或不构建）哪些工具进行谨慎、选择性的规划，能够带来丰厚回报。

为工具划分命名空间

你的 AI 智能体可能会接入数十个 MCP 服务器和数百个不同的工具——其中包括其他开发者编写的工具。当工具功能重叠或目的模糊时，智能体会难以判断该使用哪一个。

通过命名空间（按共同前缀将相关工具分组）可以帮助界定大量工具之间的边界；MCP 客户端有时会默认这样做。例如，按服务对工具命名空间（如 asana_search、jira_search）和按资源命名空间（如 asana_projects_search、asana_users_search），都能帮助智能体在正确的时间选择正确的工具。

我们发现，选择使用前缀还是后缀命名空间会对工具使用评测产生非平凡的影响。效果因 LLM 而异，我们鼓励你根据自己的评测结果选择命名方案。

智能体可能会调用错误的工具，或以错误参数调用正确的工具，或调用次数太少，或错误处理工具响应。通过有选择地实现名称能够反映任务自然划分的工具，你既减少了加载到智能体上下文中的工具数量及其描述，又能把智能体的部分推理从上下文中转移回工具调用本身，从而降低整体出错风险。

让工具返回有意义的上下文

同样地，工具实现应谨慎地只向智能体返回高信号信息。它们应优先考虑上下文相关性，而不是灵活性，并尽量避免低层级的技术标识符（例如 uuid、256px_image_url、mime_type）。诸如 name、image_url 和 file_type 这样的字段更有可能直接影响智能体的后续行动和回应。

相比晦涩的标识符，智能体在处理自然语言名称、术语或标识时表现更佳。我们发现，仅仅把任意的字母数字 UUID 转换成更有语义、可解释的语言（甚至是从 0 开始的 ID 方案），就能显著减少幻觉，从而提升 Claude 在检索任务中的精准度。

在某些情况下，智能体可能需要既能处理自然语言又能处理技术标识符的输出，哪怕只是为了触发后续的工具调用（例如 search_user(name='jane') → send_message(id=12345)）。你可以通过在工具中暴露一个简单的 response_format 枚举参数来同时支持两者，让智能体控制工具返回 “concise” 还是 “detailed” 响应（如下图所示）。

你还可以添加更多格式以获得更大的灵活性，类似 GraphQL 中可以精确选择希望收到哪些信息。以下是一个用于控制工具响应详细程度的 ResponseFormat 枚举示例：

enum ResponseFormat {
   DETAILED = "detailed",
   CONCISE = "concise"

为 token 效率优化工具响应

优化上下文质量固然重要，但优化传回给智能体的上下文数量同样关键。

对于可能消耗大量上下文的工具响应，我们建议实现分页、范围选择、过滤和/或截断的组合，并设置合理的默认参数。以 Claude Code 为例，我们默认将工具响应限制在 25,000 个 token 以内。我们预计智能体的有效上下文长度会随时间增长，但对上下文高效工具的需求仍将持续存在。

如果你选择截断响应，请务必通过有用的指引来引导智能体。你可以直接鼓励智能体采取更节省 token 的策略，例如针对知识检索任务进行多次小范围、目标明确的搜索，而不是一次广泛的搜索。同样，如果工具调用触发错误（例如在输入验证期间），你可以对错误响应进行提示工程，以清晰地传达具体且可执行的改进建议，而不是晦涩的错误代码或堆栈跟踪。

对工具描述进行提示工程

我们来到改进工具最有效的方法之一：对工具描述和规格进行提示工程。由于这些描述会被加载进智能体的上下文，它们能够共同引导智能体采取有效的工具调用行为。

在撰写工具描述和规格时，想想你会如何向团队的新成员介绍工具。将你可能隐含的上下文——例如特定的查询格式、生僻术语的定义、底层资源之间的关系——明确写出来。通过严格的数据模型清晰地描述预期输入和输出，避免含糊不清。尤其是输入参数应当命名明确：与其使用 user 参数，不如使用 user_id。

借助评测，你可以更有信心地衡量提示工程的影响。即便是对工具描述的细微改进，也可能带来显著提升。在我们对工具描述进行精确微调后，Claude Sonnet 3.5 在 SWE-bench Verified 评测上取得了最新的领先表现，大幅降低错误率并提高任务完成率。

你可以在我们的开发者指南中找到其他工具定义的最佳实践。如果你正在为 Claude 构建工具，我们还建议阅读有关工具如何动态加载到 Claude 系统提示中的内容。最后，如果你在为 MCP 服务器编写工具，工具注释可以帮助标明哪些工具需要开放世界访问或会进行破坏性操作。

展望未来

要为智能体构建高效工具，我们需要将软件开发实践从可预测的确定性模式转向非确定性模式。

通过本文介绍的迭代、以评测驱动的流程，我们发现了决定工具成功的若干共性：高效工具定义清晰、意图明确，善用智能体的上下文，可在多样的工作流中组合使用，并让智能体能直观地解决真实世界的任务。

未来，我们预计智能体与世界交互的具体机制将不断演进——包括 MCP 协议的更新以及底层 LLM 的升级。采用系统化、以评测驱动的方式改进智能体工具，可以确保随着智能体能力提升，它们使用的工具也能同步演化。
